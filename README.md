# Architecture Reasoning Trainer

A personal tool for practicing system-design / architecture reasoning. One AI call generates a
realistic scenario **and** a hidden reference solution; you write your own answer; a second AI call
judges it against the reference — specifically naming patterns you used but didn't name (e.g. you
describe "delete the row first, only insert if it was deleted" and it tells you that's
compare-and-delete / optimistic locking). Sessions are stored so you can track which named patterns
you keep missing.

Beyond evaluation it's a full **trainer**: a **Study** section generates (or lets you paste in)
Markdown deep-dives on the patterns you keep missing, and an **Artifacts** section keeps an archive
of past reference designs, generated cheat-sheets, and a pinned collection. Study content can be
generated by the API model *or* pasted in from elsewhere (e.g. a Claude subscription) for zero tokens.

## Stack
- **Backend:** FastAPI + SQLAlchemy + Postgres, Alembic migrations, deps via `uv`.
- **Frontend:** React (Vite), plain CSS. Built and served by the backend in production.
- **AI:** swappable provider — Anthropic (official SDK), or OpenAI / OpenRouter (OpenAI SDK).
- **Deploy:** Render Blueprint (`render.yaml`) — web service; Postgres hosted free on Neon.

## Choosing a model
Set `LLM_MODEL` as `provider:model_id`, and set that provider's key:

| `LLM_MODEL`                              | Key needed            |
| ---------------------------------------- | --------------------- |
| `anthropic:claude-sonnet-4-6` (default)  | `ANTHROPIC_API_KEY`   |
| `openrouter:anthropic/claude-3.5-sonnet` | `OPENROUTER_API_KEY`  |
| `openai:gpt-4o`                          | `OPENAI_API_KEY`      |

Keys are read from **env only**, never stored in the database. `GET /api/models` reports which
providers are configured; the Setup screen's model dropdown lets you switch at runtime.

## Auth & multi-user
Login is **Auth0 (server-side BFF)** with "Continue with Google". The backend runs the OIDC code
flow (Authlib); only the local `user_id` is stored in an httponly session cookie — **no tokens ever
reach the browser**. Each user owns their scenarios, sessions, and study notes.

- **Visibility:** every artifact is **private by default**; the owner can flip it to **public**
  (visible to other logged-in users — never to the open internet).
- **Key protection:** all LLM calls run server-side on the host's keys, so **only emails in
  `OWNER_EMAILS` may run AI generation** (generate / judge / AI study). Everyone else can log in,
  browse public artifacts, and import their own pasted Markdown (zero token cost). Keys live in env
  only, never the DB.
- **Local dev:** leave `AUTH0_DOMAIN` blank and the app auto-logs-in a dev owner — no tenant needed.

### Auth0 setup (production)
1. Create an Auth0 **Regular Web Application**.
2. **Allowed Callback URLs:** `{APP_BASE_URL}/api/auth/callback` (add `http://localhost:8000/api/auth/callback` for local).
3. **Allowed Logout URLs:** `{APP_BASE_URL}`.
4. Enable the **google-oauth2** social connection.
5. Copy Domain / Client ID / Client Secret into `AUTH0_DOMAIN` / `AUTH0_CLIENT_ID` /
   `AUTH0_CLIENT_SECRET`; set `APP_BASE_URL`, `OWNER_EMAILS`, and `SESSION_HTTPS_ONLY=true`.

After first login, an owner can `POST /api/admin/claim-legacy` to assign any pre-auth rows to their
account.

## Run locally (docker-compose: Postgres + API)
```bash
cp .env.example .env          # set LLM_MODEL + at least one provider key
docker compose up --build     # API on http://localhost:8000 (runs migrations on boot)
curl localhost:8000/api/models
```
The API also serves the built frontend at `:8000`. For live frontend development, run it separately:
```bash
cd frontend
npm install
npm run dev                   # http://localhost:5173, proxies /api -> :8000
```

## Run the backend without Docker
```bash
cd backend
cp .env.example .env          # point DATABASE_URL at a local Postgres
uv sync
uv run alembic upgrade head
uv run uvicorn app.main:app --reload
```

## Deploy to Render (with a free Neon Postgres)
The database is hosted on **Neon** (permanent free tier) rather than Render's managed Postgres
(which expires ~30 days after creation). No code changes — Render just runs the web service.

1. **Create the Neon database:** sign in at [neon.tech](https://neon.tech) → New Project. Copy the
   **connection string** (looks like `postgresql://user:pass@ep-xxx.neon.tech/dbname?sslmode=require`).
2. Push this repo to GitHub.
3. In Render: **New → Blueprint**, point it at the repo. It reads `render.yaml` and creates just the
   web service.
4. Set these env vars in the dashboard at apply (they're `sync: false`):
   - `DATABASE_URL` — paste the Neon connection string from step 1.
   - `AUTH0_DOMAIN` / `AUTH0_CLIENT_ID` / `AUTH0_CLIENT_SECRET` — from the Auth0 setup above.
   - `APP_BASE_URL` — your Render URL, e.g. `https://arch-trainer.onrender.com`.
   - `OWNER_EMAILS` — comma-separated emails allowed to run AI generation.
   - `LLM_MODEL` (e.g. `anthropic:claude-sonnet-4-6`)
   - `ANTHROPIC_API_KEY` (and/or `OPENAI_API_KEY`, `OPENROUTER_API_KEY`)
   - `SESSION_SECRET` (auto-generated) + `SESSION_HTTPS_ONLY=true` (from the blueprint) — leave alone.
5. Deploy. Migrations (`alembic upgrade head`) run against Neon on container start; the app is served
   at the Render URL.

## API (all routes require a logged-in user except the auth + health routes)
| Method | Path                       | Purpose                                                        |
| ------ | -------------------------- | -------------------------------------------------------------- |
| GET    | `/api/me`                  | Current user `{id,email,name,is_owner}` (open)                 |
| GET    | `/api/auth/login`          | Start Auth0 login (`?connection=google-oauth2` for Google)     |
| GET    | `/api/auth/callback`       | OIDC callback — sets the session cookie                        |
| GET    | `/api/auth/logout`         | Clear session + Auth0 logout                                   |
| POST   | `/api/.../visibility`      | Toggle private⇄public on a scenario/session/study-note (owner) |
| GET    | `/api/public/study-notes`  | Public study notes/cheat-sheets from all users                 |
| GET    | `/api/public/references`   | Public reference designs from all users                        |
| POST   | `/api/scenarios`           | Generate scenario + hidden reference; returns scenario only    |
| GET    | `/api/scenarios/{id}`      | Fetch scenario (`?reveal=true` includes the reference)         |
| POST   | `/api/sessions`            | Judge an answer; returns judgment + reference                  |
| GET    | `/api/sessions`            | Session history (paginated)                                    |
| GET    | `/api/sessions/export`     | Markdown export of all sessions                                |
| GET    | `/api/stats/pattern-gaps`  | Recurring unnamed-pattern frequency                            |
| GET    | `/api/stats/summary`       | Total sessions + average score                                 |
| GET    | `/api/models`              | Configured providers + suggested models + current default     |
| POST   | `/api/study`               | AI-generate a Markdown study deep-dive (stored)               |
| POST   | `/api/cheatsheets`         | AI-generate a Markdown cheat-sheet (stored)                   |
| POST   | `/api/study-notes`         | Import a note by pasting Markdown (no LLM call)               |
| GET    | `/api/study-notes`         | List study notes / cheat-sheets (`?kind=&pinned=`)           |
| PUT/DELETE | `/api/study-notes/{id}` | Edit / delete a note; `/pin` to pin                          |
| GET    | `/api/artifacts/references`| Reference designs from answered scenarios (no-peek)          |

Single-user tool — no auth.
